# WebUI Dashboard and WebSocket Broadcast Issues

## 1. WebSocket frames do not reach frontend

### Problem
Dashboard fields (temperature, ADC, valve, burette) show `"--"` indefinitely. The WebSocket connection is established (`ws.readyState === 1`, `connectionAlive === true`), but `onmessage` never fires for broadcast data or log events.

### Root cause
`broadcastWsEvent()` in `http_server.cpp` had several issues:
- Diagnostics used `write(STDOUT_FILENO, ...)` which was invisible after UART reinit (see Issue 2)
- No null/empty checks for frame data before sending
- Session invalidation on transient `httpd_ws_send_frame_async` errors was too aggressive
- No `ESP_LOGI`/`ESP_LOGW` diagnostics (re-entrancy-safe)

### Solution
- Replaced `write(STDOUT_FILENO)` diagnostics with `ESP_LOGI`/`ESP_LOGW` (with re-entrancy guard)
- Added null/empty checks for frame data
- Improved session invalidation with logging of fd info state
- Removed session invalidation on transient `httpd_ws_send_frame_async` errors

### Edge cases
- Session fd may become stale if client reconnects (e.g., after flash). `broadcastWsEvent` cleans stale sessions by checking `httpd_ws_get_fd_info()`.
- Multiple WS clients (max 4 sessions). Not tested.

## 2. ESP_LOG forwarding to UART unstable after serial driver reinit

### Problem
After `esp_log_set_vprintf(logVprintf)` is called at the start of `app_main()`, subsequent `write(STDOUT_FILENO, buf, n)` calls in `logVprintf` do not produce visible output in the serial monitor. `puts("DBG: ...")` lines still appear. Broadcast JSON from `serial.write()` also works. Only the forwarded ESP_LOG output is lost.

### Root cause
The `SerialReader::init()` (step 4) calls `uart_driver_install()` for the console UART. This reinitializes the UART driver, which may change the internal file descriptor mapping for `stdout`/`STDOUT_FILENO`. After reinit, `write(1, ...)` may go to a different fd than `puts()`/`fwrite()`.

### Solution
- `logVprintf()` already uses `fwrite(buf, 1, n, stdout)` + `fflush(stdout)` — confirmed working
- All `write(STDOUT_FILENO)` calls removed from `broadcastWsEvent()` (replaced with `ESP_LOGI`/`ESP_LOGW`)

### Edge cases
- Early ESP_LOG calls (before step 4 serial init) may use different UART state than later calls
- `esp_log_set_vprintf` affects ALL ESP_LOG calls — any re-entrancy (ESP_LOG called from within vprintf) is silently dropped by `pushing_` guard

## 3. `/api/logs` response buffer size marginal

### Problem
`ResponseBuffer` (2048 bytes) holds approximately 20 log entries. With longer messages (e.g., JSON payloads, stack traces), fewer entries fit. The response is silently truncated when the buffer overflows, producing invalid JSON.

### Root cause
The `append()` lambda in `api_logs_handler` stops writing when `pos >= rsp.size()`. The JSON closing braces are then written at the truncated position, resulting in unterminated strings.

### Solution
- Increased from 512 to 2048 bytes (covers 20 typical entries)
- Future: implement chunked response or dynamic buffer for large queries

## 4. Frontend render race with WebSocket connection

### Problem
`loadInitialLogs()` (HTTP fetch of `/api/logs`) and `connectWs()` (WebSocket) are both called from `initApp` on `DOMContentLoaded`. If `loadInitialLogs` completes before WS is ready, initial log entries render but subsequent WS-pushed log events may be missed during the reconnect gap.

### Root cause
No synchronization between HTTP fetch and WS connection. The frontend uses `addLogEntry(msg)` for both paths independently, but the WS might not yet be connected when the initial fetch completes.

### Solution
Not critical — messages are pushed to the same `APP_STATE.logs.messages` array regardless of source. The gap only affects the first ~1 second of operation.

## 9. `/api/logs/download` connection reset (found during curl testing)

### Problem
`GET /api/logs/download` caused `Connection reset by peer` after ~7 seconds.

### Root cause
`LogEntry entries[100]` (100 × 140 bytes = 14 KB) allocated on the stack of the HTTP handler. The HTTP server runs in the `net_owner` task with 16 KB stack. A 14 KB array left only 2 KB for function calls and other locals, causing a stack overflow.

### Solution
- Reduced array to `LogEntry entries[20]` (2.8 KB)
- Fixed `snprintf` return value handling: `snprintf` returns the would-be length, not actual bytes written. When the buffer was nearly full, `pos` was incremented past the buffer boundary, causing `httpd_resp_send()` to read past the buffer. Fixed by using `std::min(n, remaining - 1)` for actual written bytes.

## 11. WS handler — session not cleaned up on recv error (2026-07-11)

### Problem
When the client disconnects without sending a WS close frame (TCP RST/timeout), `httpd_ws_recv_frame` returns an error. The handler returned the error immediately without calling `removeSession(fd)`, leaving stale fds in the 4-slot session table. New WS connections could not be tracked. The client received no broadcasts, triggering `pingServer()` → reconnect → more stale sessions.

### Evidence
- Console: `"Server back. Reconnecting WS..."` every few seconds
- Network tab: no WS frames, only HTTP `/api/status` fallback polling
- Server log: `httpd_ws_get_frame_type: Failed to read header byte (socket FD invalid)`

### Fix
- `components/infrastructure/network/src/http_server.cpp:372-379`: Added `removeSession(fd)` cleanup when `httpd_ws_recv_frame` returns error

## 12. Forensic audit vs legacy Arduino — 22 bugs found and fixed (2026-07-11)

### Background
`@explore` agent performed a line-by-line comparison of the C++ WebUI frontend + backend against the legacy Arduino (`legacy/arduino/`) and legacy Rust (`legacy/rust/`) codebases. **22 discrepancies found.**

### CRITICAL (10)

#### C-01: Direction string case mismatch
- **Root cause**: `parseDirection()` checks `"liq_in"`/`"liq_out"` (lowercase). Frontend JS sends `"LIQ_IN"`/`"LIQ_OUT"` (uppercase).
- **Impact**: Every command with `direction` param silently uses default (`LiqIn`). Motor always goes IN regardless of UI selection.
- **Fix**: `command.cpp:70-87` — compare case-insensitively with `tolower`.
- **Files**: `command.cpp`

#### C-02: ADC cal.get response format incompatible with frontend
- **Root cause**: Handler returns `aX1000` (int `a*1000`) but JS expects float `a`. No `r_squared`, `calibrated_at`, `is_default`, `points` array. No `"status":"ok"`.
- **Impact**: Entire ADC calibration panel non-functional. All values show `--`.
- **Fix**: `sensors.cpp:58-76` — returns `a` as float, adds all missing fields.
- **Files**: `sensors.cpp`

#### C-03: ADC cal.compute OLS uses wrong y-values
- **Root cause**: OLS uses index `i` (0-4) as y instead of reference voltages `[0, -177.5, 177.5, 350, -350]`. No `r_squared` computed.
- **Impact**: Calibration coefficients are mathematically wrong.
- **Fix**: `sensors.cpp:153-178` — OLS uses `ref_mv` as x, `raw_mv` as y. R² computed.
- **Files**: `sensors.cpp`

#### C-04: Broadcast missing `spd` field
- **Root cause**: `serializeBroadcast` produces `{"brt":{"sts":"...","vl":...}}` — no `spd`. `BroadcastEvent` has no speed member.
- **Impact**: WS Events panel Speed column always `--`.
- **Fix**: `broadcast.hpp:17` — added `float speedMlMin`. `broadcast.cpp:54-56` — added `"spd":%.1f`. `main.cpp:372-374` — populate from stepper.
- **Files**: `broadcast.hpp`, `broadcast.cpp`, `main.cpp`, tests

#### C-05: Limit switch status never sent to frontend
- **Root cause**: Broadcast JSON has no limit switch data. The `limitsw` event handler in WS `onmessage` never executes.
- **Impact**: `hw-limit-full` and `hw-limit-empty` always show `--`. Safety-critical status invisible.
- **Fix**: `broadcast.hpp:18-19` — added `bool limitFull`, `limitEmpty`. `broadcast.cpp:56,64-65` — added `"full":%s,"empty":%s`. `main.cpp:373-374` — populated from `gStopFull`/`gStopEmpty`.
- **Files**: `broadcast.hpp`, `broadcast.cpp`, `main.cpp`, tests

#### C-06: `burette.cal.runSpeedSeq` is a no-op stub
- **Root cause**: `handleRunCalibration()` returns `makeAckThenResponse()` immediately. `"freqs"` param never parsed (backend looks for `"measurements"`).
- **Impact**: Speed calibration button click does nothing.
- **Fix**: `burette_cal.cpp:164-189` — implemented motor command sending (`StartCalSpeedSeq`). `command.cpp:187-200` — parse `freqs` flat array.
- **Files**: `burette_cal.hpp`, `burette_cal.cpp`, `command.cpp`, `command.hpp`, `dispatch.cpp`

#### C-07: `waitForBuretteIdle()` checks `data.brt.sts` but `/api/status` has no `brt`
- **Root cause**: `/api/status` returns flat `{"state":"idle",...}`. JS checks `data.brt && data.brt.sts === "idle"`. `data.brt` is always `undefined`. Polling never resolves.
- **Impact**: Volume calibration and speed calibration can never complete — infinite hang.
- **Fix**: `webui.hpp:627` — changed to `data.state === "idle"`.
- **Files**: `webui.hpp`

#### C-08: `burette.cal.get` response missing most fields
- **Root cause**: Handler returns `{stepsPerMl, nominalVolume}` (camelCase, 2 fields). JS expects `steps_per_ml`, `nominal_vol`, `speed_coeff`, `min_freq`, `max_freq`, `calibration_date`, `is_default` (snake_case, 7 fields).
- **Impact**: Burette calibration panel shows `--` for most values.
- **Fix**: `burette_cal.cpp:60-79` — expanded response, snake_case field names, added all missing fields.
- **Files**: `burette_cal.cpp`

#### C-09: All `burette.cal.*` responses lack `"status":"ok"`
- **Root cause**: `makeCalResponse()` produces JSON without `"status"`. Frontend checks `result.status === "ok"` on every cal response.
- **Impact**: Every calibration action silently appears to fail.
- **Fix**: `burette_cal.cpp:22` — added `"status":"ok"` to `makeCalResponse()`.
- **Files**: `burette_cal.cpp`

#### C-10: `adc.cal.get`/`save` also lack `"status":"ok"`
- **Root cause**: `handleAdcCalGet` and `handleAdcCalSave` return JSON without `"status"`.
- **Impact**: ADC calibration status never loads; saves appear to fail.
- **Fix**: `sensors.cpp:67,87-88` — added `"status":"ok"` to both handlers.
- **Files**: `sensors.cpp`

### HIGH (4)

#### H-01: ADC and Electrode display identical values
- **Root cause**: Both `hw-adc` and `hw-electrode` read `data.mv`. Legacy used `data.adc.raw_mv` for ADC and `data.mv` for electrode.
- **Status**: Deferred — broadcast has no separate raw/calibrated ADC fields yet.
- **Files**: none

#### H-02: `burette.cal.runSpeedSeq` param `"freqs"` not parsed
- **Root cause**: `parseCommand()` looks for `"measurements"` but frontend sends `"freqs"`.
- **Status**: Fixed together with C-06.
- **Files**: `command.cpp`

#### H-03: Bootstrap Icons not loaded — connection icon invisible
- **Root cause**: HTML uses `<i class="bi-wifi">` but no Bootstrap Icons CSS loaded.
- **Impact**: Connection status badge shows colored background with no icon.
- **Fix**: `webui.hpp:106` — added CDN `<link>` for Bootstrap Icons 1.11.3.
- **Files**: `webui.hpp`

#### H-04: `adc.cal.compute` response has no `"data"` wrapper
- **Root cause**: Returns flat `{"status":"ok","a":...,"b":...}`. JS reads `result.data.a` → `undefined`.
- **Impact**: Compute confirmation dialog always shows `a=0, b=0, R²=0`.
- **Fix**: `sensors.cpp:208-213` — wrapped as `{"status":"ok","data":{"a":...,"b":...,"r_squared":...}}`.
- **Files**: `sensors.cpp`

#### M-02: Temperature display — no degree symbol

- **Files**: `webui.hpp`

#### M-03: Debug panel fields (stepperDrv, buretteSteps) never in broadcast
- Deferred — requires adding debug data to broadcast or dedicated WS event type.

#### M-04: Direction radio button state never synced from backend
- Deferred — WebUI is the primary controller, direction sync via serial/BLE is edge case.

### LOW (4)

- L-01: Limit switch HTML IDs renamed (`hw-limit-min` → `hw-limit-full`) — self-consistent.
- L-02: `stallGuard.get` vs `stallGuard.getThreshold` — unused by frontend.

- L-04: `/api/status` flat format vs legacy nested `brt` — protocol design choice.
