## Root cause (by component)

### C2 — broadcastWsEvent() never reaches client [FIXED]

**Root cause (two layers):**

*Layer 1:* ESP-IDF v6.0.1 (`httpd_uri.c:337-363`) handles the WebSocket
upgrade HTTP_GET **internally** without calling the registered handler:
```c
/* If the request is websocket handshake, then do not call the uri->handler */
return ESP_OK;
```
The `ws_handler` is only invoked when the **client sends a data frame**.
In a server-push model (no client messages), the handler is NEVER called,
so `addSession()` is NEVER called, and `broadcastWsEvent()` sends to nobody.

*Layer 2:* Even if data did arrive, `wsLogCallback` emitted log JSON with
raw `\n` characters in the `msg` field, causing `JSON.parse()` to throw
`SyntaxError` in the browser. The empty `catch(err){}` swallowed the error
silently, aborting the entire `ws.onmessage` try block.

**Fix (three parts):**

*Part A — ws_handler:* Add session unconditionally on ANY frame (not just
HTTP_GET). Since the handler never sees HTTP_GET, add the session on the
first data frame instead:
```cpp
int fd = httpd_req_to_sockfd(req);
if (fd >= 0 && req->user_ctx) {
    auto* server = static_cast<HttpServer*>(req->user_ctx);
    if (!server->findSession(fd)) {
        server->addSession(fd);
    }
}
```

*Part B — ws.js:* Send a subscribe message in `ws.onopen` to trigger the
server's data frame handler:
```js
ws.send(JSON.stringify({type:'sub'}));
```

*Part C — wsLogCallback:* Escape `\n`, `\r`, `\t` in the JSON msg field:
```cpp
else if (*src == '\n') { buf[n++] = '\\'; buf[n++] = 'n'; }
```

**Full analysis:** `docs/lessons_learned/LL-044.yaml`

### C9 — Recursive ESP_LOGI in broadcastWsEvent → panic [FIXED]

**Root cause:** `broadcastWsEvent()` called `ESP_LOGI`/`ESP_LOGW` for every
broadcast iteration. Each log call → `LogBuffer.push()` → `xQueueSend` →
`log_worker` → `wsLogCallback` → `broadcastWsEvent()` — unbounded recursion.

**Fix:** All `ESP_LOGI`/`ESP_LOGD`/`ESP_LOGW` calls removed from
`broadcastWsEvent()`. The function is now fully silent:
- Early return if no valid sessions (no work to do)
- No `ESP_LOGD` on fd cleanup
- No `ESP_LOGW` on send failure
- No summary `ESP_LOGD` after the loop

The only log output from the broadcast path appears on the **serial line**
via `serial.write(sv)` in `main.cpp:421` — this path has no feedback loop
into the WebSocket callback chain.
