# Crash Report: ESP32-S3 TG1WDT Boot Loop (Corrected — v2.0)

## Verdict

- **Status:** root_cause_found
- **Root Cause:** `CONFIG_ESP_INT_WDT=y` (Interrupt Watchdog, default enabled in ESP-IDF v6) with 300ms timeout. NVS, WiFi, and BLE init operations legitimately disable interrupts for flash I/O for cumulative periods exceeding 300ms. The IWDT fires → `TG1WDT_SYS_RST` → boot loop.
- **Confidence:** high (fix verified on hardware)
- **Previous diagnosis (v1.0):** Incorrectly blamed IRAM/flash placement of `__wrap_esp_panic_handler`. The exception handler was never invoked because the IWDT is a hardware timer that fires while interrupts are masked — no exception context is created.

## Evidence Chain

**Serial log:** `logs/serial_2026-07-06_19-54-37.log` (showing cycles after logger fix)

The Saved PC is in the Xtensa exception handler because the IWDT fires while the CPU has interrupts masked (during hardware exception dispatch). The Saved PC captures the current execution point, which is `_xt_handle_exc`. The varying addresses reflect different branches within the handler depending on exactly when the WDT fires.

### Step 3: Config Analysis

`sdkconfig` generated by build (from `target/sdkconfig.h`):
```
#define CONFIG_ESP_INT_WDT 1
#define CONFIG_ESP_INT_WDT_TIMEOUT_MS 300
#define CONFIG_ESP_INT_WDT_CHECK_CPU1 1
```

The Interrupt Watchdog (IWDT) on Timer Group 1 is **enabled by default** in ESP-IDF v6 with a 300ms timeout. It fires when the CPU spends >300ms with interrupts masked.

The init sequence (after the logger fix) includes:
1. `check_heap_integrity()` — flash access
2. `Peripherals::take()` — hardware register access
3. `AdcDriver::new()` — peripheral init
4. `Led::new()` — GPIO config
5. `LimitSwitch::new()` ×2 — GPIO ISR subscription
6. `RmtStepper::new()` — RMT channel alloc
7. `EspSystemEventLoop::take()` — system event loop
8. `nvs_init()` — NVS flash I/O
9. `BleManager::new()` — BLE init
10. Thread creation (net_owner)

Steps 7-10 each involve flash I/O (NVS, WiFi calibration data) that mask interrupts. The cumulative masked time exceeds 300ms, triggering the IWDT.

### Step 4: Root Cause

The IWDT is a safety mechanism that detects when interrupts are accidentally left disabled (a common firmware bug). However, legitimate ESP-IDF operations (NVS flash writes, WiFi firmware loading, BLE controller init) disable interrupts as part of normal operation. On ESP32-S3 with the full feature set, these init steps take longer than 300ms cumulative.

## Fix

### Trivial

Disable the Interrupt Watchdog in `sdkconfig.defaults` — the firmware manages its own watchdog strategy via `disable_wdt()` (TWDT).

### File Modified

- `sdkconfig.defaults` (added at end):
  ```
  # Disable Interrupt Watchdog — NVS, WiFi, and BLE init legitimately disable
  # interrupts for flash I/O longer than the 300ms IWDT timeout, causing
  # TG1WDT_SYS_RST boot-loop. Task WDT is handled by disable_wdt() in esp_safe.rs.
  CONFIG_ESP_INT_WDT=n
  ```

## Key Insight

The IWDT (`CONFIG_ESP_INT_WDT`) is a **separate watchdog** from the TWDT (`esp_task_wdt_deinit()`). On ESP32-S3 with ESP-IDF v6, IWDT is enabled by default with 300ms timeout. The v1.0 report's complex IRAM/flash cache theory was incorrect — the IWDT fires before any exception handler can run, and the Saved PC in `_xt_handle_exc` is an artifact of the WDT catching the CPU during exception dispatch, not evidence of a flash cache issue.

## Remaining Issues

- The TWDT is still active. `disable_wdt()` calls `esp_task_wdt_deinit()` which may return non-zero on ESP32-S3. Monitor for `TWDT deinit failed` warnings.
- If a real interrupt-lockup bug is introduced in future code, the IWDT will no longer catch it. Consider adding a custom `INTERRUPT_LEVEL` assertion in strategic locations.
