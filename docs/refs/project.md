# Project Specification

> **Single source of truth** for hardware pinout, architecture, protocol, state machines, NVS layout, thread model, and error hierarchy.
>
> Firmware for a laboratory burette titrator. ESP32 microcontroller, Rust (`esp-idf-hal` std mode, ESP-IDF v6.0.1). Precision stepper-driven syringe dosing with pH/temperature sensing. Communication via USB-Serial (JSON), BLE GATT (NUS), and WiFi (REST API + SSE + WebUI). Tauri desktop application as primary host.

## Hardware

### Burette Drive (Stepper Syringe)

| Property | Value |
|---|---|
| Type | Linear stepper motor driving a glass/plastic syringe |
| Stepper driver | TMC2209 (step/dir mode, UART config deferred) |
| Microstepping | 16 (default), configured via TMC2209 hardware defaults |
| Step pin | GPIO25 (RMT channel 0, TX) |
| DIR pin | GPIO26 (PinDriver::output) |
| EN pin | GPIO27 (PinDriver::output, active LOW) |
| Step frequency | 30-3000 Hz (period 33333-333 us) |
| Acceleration | Trapezoidal pre-computed ramp |
| Limit FULL | GPIO32 (pull-down, pos-edge interrupt) - syringe bottom, burette full |
| Limit EMPTY | GPIO35 (floating, pos-edge interrupt) - syringe top, burette empty |

### Valve

| Property | Value |
|---|---|
| Type | 2-way solenoid valve (input/output) |
| VALVE pin | GPIO14 (PinDriver::output) |
| Positions | `input` (LOW, draw from titrant bottle), `output` (HIGH, dispense into vessel) |

## Stepper Motor Control

### RMT-Based Pulse Generation

Step pulses are generated by the ESP32 RMT peripheral (channel 0, TX mode, 1 MHz resolution). Each step = 2 RMT symbols: 1 us HIGH pulse + (interval - 1) us LOW. The RMT hardware generates the pulse train with zero CPU load per pulse.

```
RMT resolution: 1 MHz (1 tick = 1 us)
Pulse width:    1 tick HIGH
Chunk size:     128 symbols per transmit
Transmit mode:  send_and_wait() in dedicated motor thread
```

### Motion Profile

- Trapezoidal acceleration (pre-computed, host-testable)
- No S-curve needed for titrator application
- `RampConfig`: accel_steps, decel_steps, min_interval_us (full speed), max_interval_us (start/stop)
- `compute_ramp()` yields a `Vec<u32>` of per-step intervals in us

### Channel-Based Communication

```rust
// domain/channels.rs
pub struct SystemChannels {
    pub cmd_tx: Sender<BuretteCommand>,
    pub cmd_rx: Receiver<BuretteCommand>,
    pub status_tx: Sender<StatusUpdate>,
    pub status_rx: Receiver<StatusUpdate>,
    pub log_tx: Sender<LogEntry>,
    pub log_rx: Receiver<LogEntry>,
}
```

Motor thread reads `cmd_rx`, writes atomics. Main loop polls atomics, publishes status. BLE notify thread reads `status_rx`, calls NimBLE `notify()`.

### Thread Architecture

```
MOTOR THREAD (prio 20, 4 KB stack):
  std::thread. Owns RmtStepper exclusively.
  loop { read atomic target -> send_and_wait -> write atomic position }
  send_and_wait() blocks only this task, NOT main loop

MAIN LOOP (prio 10, 8-16 KB stack):
  loop {
    cmd_queue.process()           -> set motor target atomics
    wifi_process()                -> DNS poll, reconnect logic
    sse_push()                    -> non-blocking socket write
    temperature_poll()            -> DS18B20 non-blocking
    adc_poll()                    -> 64 samples @ 1ms spacing
    led_process()                 -> blink state machine
    transport_sm()                -> USB / BLE priority handoff
    sleep(10ms)                   -> pacing tick
  }

TEMPERATURE THREAD (prio 15, 16 KB stack):
  loop {
    OneWire bitbang sequence
    store result in AtomicI32
    sleep(1000ms)
  }

BLE NOTIFY THREAD (prio 15, 8 KB stack):
  loop { recv mpsc -> nimble_notify }
```

**Golden Rule:** The main loop must NEVER execute a blocking operation (`send_and_wait`, `lock()`, `recv()`). All blocking calls live in dedicated threads.

## Communication Protocol

### Transport Priority

USB has absolute priority over BLE. The transport state machine (`transport_sm()`) runs every main loop tick:

```
USB alive?        -> switch to USB, disconnect BLE, stop advertising
BLE alive?        -> switch to BLE, stop advertising
Neither alive?    -> fallback to USB, start BLE advertising
```

## State Machine

### Burette State Machine

```
Idle -> Homing -> Idle (on success)
Idle -> Filling -> Idle
Idle -> Emptying -> Idle
Idle -> Dosing -> Idle
Idle -> Rinsing -> Idle
Any -> Error (on fault)
Any -> Stopping -> Idle (on stop)
```

```rust
pub enum BuretteState {
    Idle,
    Homing,
    Filling { target_ml: f32 },
    Emptying { target_ml: f32 },
    Dosing { remaining_ml: f32 },
    Rinsing { phase: RinsePhase, cycles_left: u32 },
    Stopping,
    Error,
}
```

### Transport State Machine

```
USB_ACTIVE -> TRANSPORT_USB
BLE_ACTIVE (no USB) -> TRANSPORT_BLE
NO_ACTIVE -> TRANSPORT_USB (fallback) + BLE advertising
```

## Safety Interlocks

| Rule | Behaviour |
|---|---|
| Limit switch during motion | RMT emergency_stop(), motor disabled, error state |
| Concurrent commands | BUSY response if burette SM is moving |
| USB takeover | BLE disconnected immediately; in-flight command continues to completion |
| BLE zombie defence | 5 consecutive notify failures -> disconnect + restart advertising |
| WDT | Disabled at boot (`esp_task_wdt_deinit()`) - RMT blocks >250 ms |

## Error Handling

Three-level typed hierarchy with automatic `?` conversion:

```
AppError
 +-- Hardware(HardwareError)
 |    +-- StepperMotor(StepperError)
 |    |    +-- InitFailed { reason }
 |    |    +-- Rmt { code }
 |    |    +-- LimitSwitchTriggered { switch }
 |    |    +-- Timeout { operation }
 |    +-- Sensor(SensorError)
 |    |    +-- AdcReadFailed
 |    |    +-- TempSensorNotDetected
 |    |    +-- TempReadGlitch
 |    +-- Network(NetworkError)
 |         +-- WifiConnectionFailed
 +-- Protocol(ProtocolError)
 |    +-- InvalidJson
 |    +-- UnknownCommand
 |    +-- MissingParam
 +-- State(StateError)
 |    +-- Busy
 |    +-- InvalidTransition
 |    +-- AlreadyRunning
 +-- Resource(ResourceError)
      +-- NvsOpenFailed
      +-- OutOfMemory
```

All errors: logged via the `log` crate (ring buffer, 100 entries), reported in JSON responses (`AppError::to_json()`), broadcast via SSE.

### Recovery Strategy

Implement `Recoverable` trait for error types:

| Error | Recoverable | Action |
|---|---|---|
| `NetworkError` | Yes | Retry after STA_RECONNECT_INTERVAL |
| `StepperError::Rmt` | Yes | Retry chunk transmission |
| `StepperError::Timeout` | No | Enter error state, require manual reset |
| `StateError` | No | Fix caller logic |
| `ProtocolError` | No | Log and ignore malformed command |

### Strict Technology Stack & Constraints

**Target:** `xtensa-esp32-espidf` (ESP-IDF v6.0.1 managed via embuild git master).

**Framework:** `esp-idf-hal` (std mode), **not** `esp-hal` (no_std). We need WiFi, BLE, NVS, std threads.

**RMT Stepper:** Use `TxChannelDriver` + `CopyEncoder` for pulse generation. NOT `Ets::delay_us` busy-wait. RMT runs in a dedicated motor thread (4 KB stack).

**Concurrency Model:**
- Main loop: non-blocking polling (atomics, `try_lock`, 10 ms tick)
- Motor thread: blocking RMT transmits, 4 KB stack
- Temperature thread: blocking OneWire bitbang, 16 KB stack
- BLE notify thread: blocking `mpsc::recv`, 8 KB stack
- HTTP server: `EspHttpServer` internal task, 12 KB stack

**Async:** Do NOT use async/await. All concurrency is via `std::thread` + atomics + `mpsc` channels.

**WiFi:** `EspWifi` + `BlockingWifi` from `esp-idf-svc`. Captive portal AP with custom `EspNetif` (IP 192.168.4.1/24). DNS responder via `UdpSocket`.

**HTTP Server:** `EspHttpServer` with `stack_size: 12288`. SSE via raw socket fd (`httpd_resp_send_chunk`).

**BLE:** `esp32-nimble` (local patched for IDF v6). Patched `cfg_if` blocks in `ble_characteristic.rs` for `esp_idf_version_major = "6"`.

**JSON:** `serde` + `serde_json`. Hot-path serialization uses `json!()` macro (fixed alloc). Avoid `serde_json::Value` for dynamic parsing -- use streaming deserializer or `json!()`.

**Storage:** `esp_idf_sys::nvs` FFI for WiFi credentials. Burette calibration stored in NVS namespace.

**Logging:** `log` crate + `EspLogger` (std mode). Ring buffer (heapless::Deque, 100 entries) for HTTP log access.

**Endstops:** Interrupt-driven (`PosEdge` ISR -> `AtomicBool`), polled in main loop. Active HIGH.

**WDT:** Disabled at boot (`esp_task_wdt_deinit()`). RMT `send_and_wait()` blocks > 250 ms.

**Memory Budget:** All hot paths use fixed-size `heapless` buffers defined in `domain::memory`:

| Buffer | Size | Used by |
|---|---|---|
| `CommandBuffer` | 256 bytes | Serial + BLE + HTTP command input |
| `ResponseBuffer` | 512 bytes | JSON response serialization |
| `LogBuffer` | 100 entries | Ring-buffer log for HTTP access |
| `AdcBuf` | 64 x u16 | ADC rolling average |

`Vec`/`String` are **only** allowed in init code and config-change paths (e.g., `compute_ramp()`).
